2.       Concept of Operations

`Wi-Fi-based Indoor Position System (WIPS)’ aims  to track the position changes of mobile facilities based on the Wi-Fi finger print detection.  It will facilitate its users to position the mobile device through the Wi-Fi fingerprints. WIPS has one UI  to register a mobile device,  one UI to import the information of Wi-Fi hotspots and one UI to import maps on which Wi-Fi hotspots and registered devices are shown. For more details on the usage and capabilities of the application read the section,System Capabilities.

2.2 System capabilities

Wi-Fi-based Indoor Position System is a software project to track the position changes of mobile facilities based on the Wi-Fi finger print detection. Its abbreviation is WIPS. The system will run on a mobile platform, such as Android. WIPS has a UI to register a mobile device and import the information of Wi-Fi hotspots. When a user wants to use the system, he must first register his device in the system.

After the user completes the device registration, the system will import the Wi-Fi hotspot information. When a mobile device tries to connect to a Wi-Fi hotspot, information exchange occurs, which is called Wi-Fi fingerprint. The system will collect the Wi-Fi fingerprint information. If there is an error in the work, the system will not proceed to the next step, but will try to repeat the work until it is completed successfully. When these preparations are completed, the system will display a map of Wi-Fi hotspots and registered devices on its UI. After seeing the map, users can confirm that they have successfully registered and that the system can work normally.

The system can calculate the location of the device according to the Wi-Fi fingerprint. If the device is in the building, the system will draw the internal structure of the building, and then use the Wi-Fi hotspots inside the building to calculate the location of the device. The location information of the device will be marked on the map of the system. After calculating the location information, the system will save the information, so that users can query the history at any time.

After calculating the location information of the device many times, the system can connect each location point and show it to the user. In other words, by drawing a mobile path for a device, the system allows users to track it. After system analysis, the system can also predict its location according to its moving paths.

All functions are guaranteed to run correctly by the system administrator. The system administrator will monitor the data information of various functions in the background. The whole system will be perfect, it has reliable algorithm and excellent background staff. Even if there are errors, they will be solved in the shortest time.

Internal Requirements

Module: algorithm module--#include “all.h”

In the module of algorithm, we aim to enable WIPS to give a specific location of users. In this module, we need to read the wifi information storage path of each map from the database module. We need wifi information file from the module of Server.And besides, we rely on the module of Server to provide us MAC address and the signal strength of wifi selected from current position. In this way, we use all the information to give the specific location of users to the module of server. The positioning accuracy is about 5 meters

Wi-Fi Fingerprint - Pigeon Farm

The use of Wi-Fi to estimate location is a great approach, since Wi-Fi Access Points(AP) are readily availbale in large quantities in today's indoor environment and it is possible to use available mobile devices on the user's side.

 

The key part of the whole process is the Wi-Fi Fingerprint acquisition - the module we are working on. Based on the information we obtain, specific algorithm will be able to calculate the accurate position of the user.

 

The following picture clearly shows modules which are related to Wi-Fi Fingerprint and the relationship between them.

 

 

Aims of Module Wi-Fi Fingerprint are as follows:

1.              Provide an interface for Module Client. Whenever a user position requirement is activated, get Wi-Fi Fingerprint infomation and return it, so that Module Client will be able to send it to Module Algorithm.

2.         Provide an interface for Module Server. Whenever the admin requires the current Wi-Fi Fingerprint,  get Wi-Fi Fingerprint infomation back to Module Server.

Module: Map by Joy Division

In this module, our goal is to provide a series of interface fuctions for server, so that they can call them and get user's current position or user's trace. Meanwhile, we need data from the database. Specifically, map and coordinates. The coodinates are stored in table, while maps are stored on server. In general, we need to get data from client/web page, integrate them with the map from server, to form a  picture, and provide the picture to the client/web page directly. The format of  'picture' is called 'TileLayer' in Baidu Map API, whose samples are provided by customer(Prof. Mestere).

 

The following picture clearly shows modules which are related to Map and the relationship between them.

 

Aims of Module Map are as follows:

1.              Get map picture and coordinates of  points to be shown.

2.              Add the picture as a layer onto the API, and mark out users position/track.

3.              Provide  integrated map to client/web page.

 

 

Module: Server by Binary Doodles

 

The role of our module is to process request

from client and return the result to client.

 

For example, in position function, we need

Algorithm module to calculate location and we need to save location to database,

then return the location to the client.

 

Aim of our module is to provide service on the Linux server except Algorithm and database and return results to the front

end user or admin.

 

Module: Database by Minimum Debug 

The key responsibility of the database module is to establish, organize and maintain the data. For other modules, they can add,  update, delete and search for the data if neccesary. For instance, we will obtain a series of data from the Wi-Fi Fingertip Module, and then we will make the tranfer job for the future use of the Algorithm Module. The following figure shows the basic relationship between Database Module and other modules.

 

Module: Client by Hiya

The key responsibility of the client module is to send requests to the server and receive the result and show it on the UI. For example, if the user wants to know the past position, the client will send a positioning request and the past time to the server. Then the server connects with other module and gets a result. Finally the client module receives the result and the user will see it on the UI.

 

Aim of our module

1. Provide services to users, including register, login, logout, managing account, importing maps, checking positions and tracking deivces.

2. Receive the requests of the users and send them to the server.

3. Get the result of the request from the server and show it to the users.

 

4.  Quality Requirements

The application must be competitive with similar applications in regards to performance, reliability, consistency, and scalability.

 

Performance:

 

Because GPS, and equivalent technologies, do not work inside buildings, in these environments alternative technologies must be used. These solutions include those that are based on wireless networks, such as WiFi. Because WiFi is ubiquitous in all buildings of interest for this application, this will the technology on which the positioning system must be implemented. Because the exact position of all the WiFi Access Points is not known, as location estimation technique, Fingerprinting must be used. As GPS positioning is not good enough inside edifices, WIPS is a solution to facilitate people to position the mobile device through the Wi-Fi fingerprints.

4.    Glossary

 

Source

Source means where the data come from in each use case.

 

 

Administrator’s responsibility

Administrator ought to collect enough WIFI fingerprints, which will be used in device location. Also, he needs to draw the inner structure of a building, and save it as inner map data. Besides, it is his duty to make sure the system running normally. Administrator should protect user information from being stolen too.

 

Map adjustment

When administrator finishes drawing the inner structure of a building, he needs to adjust the map to make it look better. He interacts with a GUI providing several adjustment parameters. When he feels the preview of the map is nice, he will choose to save the map.

 

Map data

-Inner structure: The paths, WIFI hotspots and stores or restaurants inside the building. It also includes elevators when the building is multi-story. It is drawn by administrator of the application.

 

-Outer structure :The shape of the building. It can be accessed from GPS or drawn by administrator. Map outside the building may not be shown.

 

Query position

Query position means that users can use the software to know the position of the devices currently.

 

Historical position

Historical position means that users can use the software to know the position of the devices at a certain time in the past.

 

Register

Register is the first necessary thing users should do before using the software. Users need to add a new specific device.

 

 

Wi-fi fingerprint

Because Wi-fi is ubiquitous in all buildings of interest for this application, we will implement the positioning system based on this technology. Therefore, because the exact position of all the Wi-fi Access Points is not known, we will use a technique named Wi-fi fingerprinting as a location estimation technique. The administrator is responsible for the management of the fingerprint data collection.

The user can scan for Wi-fi fingerprints and add them to training data.
